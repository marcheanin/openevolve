## 1. Идея и мотивация

### 1.1 Проблема

Задача классификации тональности (sentiment analysis) отзывов на товары Amazon в категории **Home & Kitchen** по шкале 1–5 звёзд — нетривиальная задача даже для современных LLM. Основные сложности:

- **Субъективность оценок**: граница между 4 и 5 звёздами, между 2 и 3 — размыта и зависит от стиля рецензента.
- **Сарказм и ирония**: положительные слова, используемые в отрицательном контексте.
- **Обновления отзывов**: рецензент может изменить мнение в том же тексте.
- **Fairness**: необходимо обеспечить одинаково хорошую работу для всех пользователей, а не только для «типичных».

Ручное создание промпта, который обрабатывает все эти нюансы, — чрезвычайно трудоёмкий процесс. Каждое улучшение для одних паттернов может ухудшить работу для других.

### 1.2 Основная идея

Объединить **эволюционную оптимизацию кода** (OpenEvolve) с **активным обучением** (Active Learning) для автоматической итеративной оптимизации промптов LLM. Вместо того чтобы человек вручную подбирал правила и примеры, система:

1. Автоматически находит «сложные» примеры (Hard set), на которых ансамбль моделей ошибается.
2. Эволюционно оптимизирует промпт, фокусируясь на этих сложных случаях.
3. Переоценивает данные, выявляя новые сложные паттерны.
4. Повторяет цикл, постепенно улучшая промпт.

Ключевое отличие от обычного prompt engineering — **полностью автоматический, замкнутый цикл** без участия человека в процессе оптимизации.

## 2. Технологический стек

### 2.1 OpenEvolve — эволюционный движок

**OpenEvolve** — open-source фреймворк для эволюционной оптимизации кода с помощью LLM. Создан по мотивам AlphaEvolve (Google DeepMind). Ключевые компоненты:

| Компонент | Описание |
|-----------|----------|
| **MAP-Elites** | Алгоритм quality-diversity: поддерживает архив решений, разнообразных по выбранным признакам (feature dimensions). Каждая «ячейка» в сетке признаков хранит лучшее решение для данной комбинации. Это предотвращает схлопывание в одну точку оптимума. |
| **Island Model** | Параллельные «острова» эволюции с периодической миграцией лучших особей между ними. Предотвращает преждевременную конвергенцию. |
| **LLM-мутатор** | LLM генерирует мутации (новые варианты промпта) на основе родительского промпта, топ-решений из архива и артефактов ошибок. |
| **Evaluator** | Оценивает каждый кандидат (промпт) и возвращает скоры + артефакты (примеры ошибок для обратной связи). |
| **Controller** | Оркестрирует цикл: выбор родителя → мутация → оценка → обновление архива. |

В данном проекте OpenEvolve использован нестандартно: вместо оптимизации исполняемого кода, он **оптимизирует текстовый промпт** для классификации.

### 2.2 Датасет WILDS Amazon Reviews

Используется бенчмарк **WILDS** (Stanford) — набор данных Amazon Reviews с метаинформацией о пользователях.

| Параметр | Значение |
|----------|----------|
| Категория | Home & Kitchen (id=24) |
| Шкала оценок | 1–5 звёзд |
| Разбиение | 70% train / 15% val / 15% test |
| Seed | 42 |
| User-disjoint | Да (пользователи в train/val/test не пересекаются) |
| Стратификация | По классам |
| Min reviews per user | 10 |
| Max train users | 25 |
| Max val users | 15 |
| Итого примеров (train) | ~513 (используемый пул) |
| Итого примеров (val) | ~181 |
| Итого примеров (test) | ~208 |

Условие **user-disjoint** критически важно: модель должна обобщать на новых пользователей, а не запоминать стиль конкретных рецензентов.

### 2.3 Ансамбль LLM-воркеров

Для оценки промпта используется ансамбль из **трёх моделей** (majority vote):

| Модель | Назначение | Параметры |
|--------|-----------|-----------|
| DeepSeek Chat V3 | Воркер #1 | temperature=0.1, max_tokens=64 |
| Gemma 3 27B | Воркер #2 | temperature=0.1, max_tokens=64 |
| GPT-4o-mini | Воркер #3 | temperature=0.1, max_tokens=64 |

Каждый воркер получает один и тот же промпт с подставленным текстом отзыва и возвращает рейтинг 1–5. Итоговый ответ — **мажоритарное голосование**. Ансамбль обеспечивает устойчивость оценок: если 2 из 3 моделей согласны, ответ стабилен.

### 2.4 Модель-эволютор

Для генерации мутаций промпта используется **DeepSeek R1** (deepseek/deepseek-r1) через OpenRouter:

| Параметр | Значение |
|----------|----------|
| Temperature | 0.8 (высокая для разнообразия мутаций) |
| Max tokens | 8192 |
| Timeout | 240 сек |
| Retries | 3 |

R1 выбран за сильные способности к reasoning — он анализирует ошибки и генерирует целевые правила.

## 3. Архитектура системы

### 3.1 Двухуровневая эволюция

Система реализует **двухуровневую эволюционную оптимизацию**:

```
┌──────────────────────────────────────────────────────────────┐
│                    УРОВЕНЬ 2: Active Learning Loop           │
│                    (active_loop.py)                           │
│                                                              │
│  Цикл 0 ──► Цикл 1 ──► Цикл 2 ──► ... ──► Цикл N          │
│     │           │           │                   │            │
│  [Эволюция] [Эволюция] [Эволюция]          [Эволюция]       │
│  [Реклас.]  [Реклас.]  [Реклас.]            [Реклас.]        │
│  [Консолид.][Консолид.][Консолид.]          [Консолид.]      │
│  [Валидация][Валидация][Валидация]           [Тест]           │
│                                                              │
│ ┌──────────────────────────────────────────────────────────┐ │
│ │              УРОВЕНЬ 1: OpenEvolve Цикл                  │ │
│ │              (15–30 итераций)                             │ │
│ │                                                          │ │
│ │  Родитель ──► LLM мутация ──► Оценка ──► MAP-Elites      │ │
│ │      ▲                            │                      │ │
│ │      └────── Артефакты ◄──────────┘                      │ │
│ │              (ошибки, покрытие правил)                    │ │
│ └──────────────────────────────────────────────────────────┘ │
└──────────────────────────────────────────────────────────────┘
```

### 3.2 Структура промпта

Промпт имеет XML-структуру с чётким разделением зон ответственности:

```xml
<System>
  <Role>         <!-- Заморожен: роль модели -->
  <BaseGuidelines>  <!-- Заморожен при эволюции, модифицируется при консолидации -->
  <DynamicRules>    <!-- Эволюционируется: правила классификации для каждого уровня -->
</System>

<FewShotExamples>   <!-- Эволюционируется: примеры отзывов с рейтингами -->

<Task>              <!-- Заморожен: шаблон ввода -->
```

**DynamicRules** — основная область оптимизации. Содержит:
- Критерии для каждого уровня звёзд (1–5) с конкретными индикаторами.
- Пограничные случаи (edge cases) для разрешения неоднозначностей.
- Общее число правил ограничено ~50 для предотвращения «раздувания» промпта.

**FewShotExamples** — набор из 7–10+ примеров, покрывающих все 5 уровней рейтинга. Приоритет отдаётся часто путаемым парам (4 vs 5, 2 vs 3).

## 4. Детальное описание пайплайна

### 4.1 Инициализация

1. **Загрузка данных**: из WILDS загружается категория Home & Kitchen, разбивается на train/val/test с seed=42 и user-disjoint разбиением.
2. **Начальный прогон**: стартовый промпт (hand-crafted, ~91 строка, ~1207 токенов) прогоняется на всём train-пуле через ансамбль.
3. **Классификация примеров**:
   - **Hard**: примеры, на которых ансамбль ошибся, ИЛИ воркеры сильно не согласны (disagreement > threshold).
   - **Anchor**: примеры, на которых ансамбль правильно ответил И все воркеры согласны.
4. **Baseline тест**: стартовый промпт оценивается на test-сплите для фиксации отправной точки.

### 4.2 Цикл активного обучения (один AL-цикл)

#### Шаг 1: Построение активного батча

`DataManager.build_active_batch()` формирует батч ровно из **80 примеров**:
- **70% Hard** (~56 примеров): выбираются с использованием K-Means кластеризации для максимального разнообразия.
- **30% Anchor** (~24 примера): стратифицированная выборка по меткам для представительности.
- Если Hard или Anchor не хватает, добирается из оставшихся Seen-примеров.

#### Шаг 2: Эволюция (OpenEvolve)

OpenEvolve запускается на **15–30 итераций** с фиксированным активным батчем:

1. **Выбор родителя**: из MAP-Elites архива (сетка по `prompt_length` × `Acc_Hard`, 5×5 бинов).
2. **Мутация**: DeepSeek R1 получает:
   - Родительский промпт
   - Топ-5 лучших промптов из архива
   - Артефакты ошибок (конкретные примеры с gold/predicted/votes)
   - Покрытие правил (какие категории 1–5 покрыты в DynamicRules)
3. **Оценка** (`evaluate_fast`): кандидат прогоняется на активном батче через ансамбль.
4. **Фитнес-функция**:

```
fitness = 0.5 × Acc_Hard + 0.3 × Acc_Anchor + 0.2 × kappa_Hard − length_penalty
```

Где:
- `Acc_Hard` — точность на Hard-подмножестве батча (вес 0.5 — главный приоритет).
- `Acc_Anchor` — точность на Anchor-подмножестве (вес 0.3 — не допускать регрессии).
- `kappa_Hard` — Cohen's kappa между воркерами на Hard (вес 0.2 — согласованность ансамбля).
- `length_penalty` — 0.02 за каждые 100 токенов сверх 2000 (борьба с раздуванием промпта).

5. **MAP-Elites архив**: кандидат размещается в ячейку по длине промпта и Acc_Hard. Если ячейка занята менее сильным решением — замена.

#### Шаг 3: Переклассификация

После эволюции батч переоценивается лучшим промптом. Примеры переклассифицируются:
- Ранее Hard, теперь правильно и согласованно → перемещаются в Anchor.
- Ранее Anchor, теперь неправильно → перемещаются в Hard.

Это обеспечивает динамическое обновление сложности задачи.

#### Шаг 4: Валидация

Лучший промпт оценивается на **полном validation-сплите** (~181 пример). Метрики:
- `R_global`: общая точность.
- `R_worst`: точность на худшей 10-й перцентили пользователей (fairness-метрика).
- `MAE`: средняя абсолютная ошибка.
- `Acc_Hard` / `Acc_Anchor`: по текущей Hard/Anchor классификации val-сплита.
- `mean_kappa`: средний Cohen's kappa между воркерами.
- `combined_score`: `0.4×R_global + 0.3×R_worst + 0.3×(1 − MAE/4)`.

#### Шаг 5: Консолидация (Уровень 2)

Между циклами запускается **консолидация** — мета-оптимизация:

1. Берутся топ-3 промпта из MAP-Elites архива текущего цикла.
2. DeepSeek R1 получает лучший промпт как базу, топ-3 как reference, артефакты ошибок.
3. В отличие от эволюции, консолидация **может модифицировать все секции** промпта, включая `<BaseGuidelines>`.
4. Задача: объединить лучшие правила из разных промптов, удалить дубликаты, сохранить покрытие всех категорий.

**Consolidation Gate**: консолидированный промпт оценивается на **validation-сплите** (не на батче!) и принимается, только если:

```
cons_val_combined + 0.02 ≥ evo_val_combined
```

Т.е. допускается просадка не более 0.02 по combined_score. Иначе остаёмся на лучшем промпте эволюции.

#### Шаг 6: Pool Refresh & Expansion

**Refresh** (каждый цикл): 10 новых примеров из Unseen-пула добавляются в Hard pool (`refresh_per_cycle=10`). Это обеспечивает постоянный приток свежих данных.

**Hard Expansion** (триггерная): если `n_hard ≤ 15` — пул расширяется из Unseen.

**Soft Expansion** (по стагнации): если за 3 цикла (`soft_expansion_patience=3`) не было улучшения `evo_best_score` на ≥ 0.02 (`soft_expansion_min_delta`), пул расширяется. Новые примеры выбираются **семантически далёкие от Anchor** (по косинусному расстоянию), чтобы добавлять максимально информативные случаи.

#### Шаг 7: Тест

В конце каждого цикла промпт оценивается на **test-сплите** (~208 примеров) для отслеживания реальной генерализации. Эти метрики не влияют на эволюцию — только для мониторинга.

### 4.3 Трекинг лучшего промпта

Глобально отслеживается `best_val_prompt` — промпт с наивысшим `val_combined_score` за всю историю AL. Сохраняется в `best_val_prompt.txt`. Это страховка от деградации: если последующие циклы ухудшают промпт, всегда доступен лучший найденный вариант.

### 4.4 Артефакты обратной связи

При каждой оценке кандидата формируются **артефакты ошибок** для LLM-мутатора:

- **ERROR DISTRIBUTION**: матрица ошибок (gold → predicted), наиболее частые пары.
- **MISCLASSIFIED EXAMPLES**: полный текст отзыва + gold rating + predicted + голоса каждого воркера + disagreement score.
- **BORDERLINE EXAMPLES**: правильный ответ, но воркеры не согласны между собой.
- **RULE COVERAGE**: какие категории 1–5 имеют правила в DynamicRules, а какие отсутствуют.

Это обеспечивает **целевую обратную связь**: мутатор видит конкретные ошибки и может создать правила именно для них.

### 4.5 Механизм возобновления (Resume)

Система поддерживает **graceful recovery** при сбоях API:
- При ошибке 401/429 и других API-ошибках сохраняется `resume_state.json`.
- При перезапуске с `--resume-from-dir` система восстанавливает состояние и продолжает с того же цикла.
- Также поддерживается **расширение** завершённого эксперимента: если все циклы завершены, можно добавить новые, указав `--n-al` больше текущего количества.

## 5. Эксперименты

### 5.1 Предшествующие эксперименты (Exp 1–3b): базовый подход

До активного обучения были проведены серии экспериментов с **прямой эволюцией** промпта (без AL-цикла):

| Эксперимент | Описание | R_global | R_worst | MAE | Combined |
|-------------|----------|----------|---------|-----|----------|
| Exp 1 | 1 модель (gpt-oss-120b), без эволюции | 51.02% | 31.03% | 0.704 | 0.544 |
| Exp 2 | 1 модель + эволюция OpenEvolve | 54.61% | 34.55% | 0.640 | 0.574 |
| Exp 3a | Ансамбль 3 моделей, без эволюции | 65.13% | 44.01% | 0.362 | 0.717 |
| Exp 3b | Ансамбль 3 моделей + эволюция OpenEvolve | 68.27% | 47.69% | 0.338 | 0.736 |

**Выводы из ранних экспериментов:**
- Эволюция даёт +3–3.5% R_global как для single model, так и для ансамбля.
- Ансамбль сам по себе даёт значительно больший прирост (+14%).
- Проблема: эволюция без AL быстро «наедается» — улучшения после первых итераций минимальны.

### 5.2 Эксперимент v2: первый Active Learning

Первый прогон с активным обучением — **5 AL-циклов**, 30 итераций эволюции на цикл.

| Цикл | val_R_global | val_R_worst | val_combined | evo_best | Время |
|------|-------------|-------------|--------------|----------|-------|
| 0 | 76.80% | 53.33% | 0.838 | 0.749 | 44 мин |
| 1 | 75.69% | 54.00% | 0.835 | 0.691 | 69 мин |
| 2 | 74.03% | 44.00% | 0.799 | 0.762 | 3 мин |
| 3 | 75.14% | 44.00% | 0.802 | 0.715 | 43 мин |
| 4 | 75.14% | 35.00% | 0.776 | 0.666 | 14 мин |

**Проблемы v2:**
- **Деградация** по val_combined_score: с 0.838 до 0.776.
- **Нестабильность**: R_worst упал с 53% до 35%.
- **Отсутствие механизмов контроля**: нет consolidation gate, нет pool refresh, нет ограничения длины промпта.
- Цикл 2 длился всего 3 минуты — скорее всего, произошёл сбой или раннее завершение.

### 5.3 Эксперимент v3: полноценный Active Learning (основной)

Основной эксперимент: **10 AL-циклов**, с полным набором улучшений.

#### 5.3.1 Конфигурация v3

| Параметр | Значение |
|----------|----------|
| AL-циклов | 10 (0–9) |
| Итераций эволюции | 30 на цикл (80 в цикле 0) |
| Batch size | 80 (фиксированный) |
| Hard ratio | 70% |
| Population size (MAP-Elites) | 30 |
| Feature dimensions | prompt_length × Acc_Hard |
| Feature bins | 5×5 |
| Prompt length limit | 2000 токенов |
| Length penalty | 0.02 / 100 токенов |
| Soft expansion patience | 3 цикла (в config стоит 5, в финальной v4 — 3) |
| Refresh per cycle | 10 примеров |
| Expansion trigger | 15 hard |
| Consolidation model | DeepSeek R1 |

## 6. Ключевые технические решения и фичи

### 6.1 MAP-Elites для промптов

Стандартный MAP-Elites работает с числовыми оптимизируемыми объектами. Адаптация для промптов:
- **Feature dimension 1**: `prompt_length` — длина промпта в токенах. Обеспечивает разнообразие по компактности: архив хранит и короткие, и длинные решения.
- **Feature dimension 2**: `Acc_Hard` — точность на сложных примерах. Обеспечивает разнообразие по фокусу: некоторые промпты лучше на одних подмножествах, другие — на других.
- **5×5 бинов**: 25 ячеек в архиве, каждая хранит лучшее решение для своей комбинации длины и точности.

### 6.2 Weighted Fitness с приоритетом Hard

Формула fitness сознательно отдаёт 50% веса Acc_Hard, что заставляет эволюцию фокусироваться на улучшении «сложных» случаев. 30% отдано Acc_Anchor как защита от регрессии: промпт не должен «ломать» то, что уже работает. 20% — на kappa_Hard для стимулирования согласованности ансамбля.

### 6.3 Consolidation Gate на Validation

Ключевая защита от деградации. Консолидированный промпт может выглядеть хорошо на batch, но терять на validation. Gate гарантирует, что принимаются только консолидации, не ухудшающие generalization более чем на 0.02.

### 6.4 Adaptive Pool Management

Три механизма управления пулом обеспечивают постоянный вызов для эволюции:
- **Hard expansion**: когда Hard-примеров слишком мало, добавляются новые из Unseen.
- **Soft expansion**: при стагнации (3 цикла без улучшения) — принудительное расширение.
- **Refresh**: постоянный приток 10 новых примеров каждый цикл.

Примеры из Unseen выбираются **семантически далёкие от Anchor** — т.е. наиболее отличающиеся от «лёгких» случаев, что максимизирует информативность расширения.

### 6.5 Rule Coverage Warning

В артефактах для LLM-мутатора есть секция RULE COVERAGE, которая предупреждает, если для какой-то категории (1–5) отсутствуют правила в DynamicRules. Это предотвращает «забывание» категорий при эволюции.

### 6.6 Length Penalty

Промпты имеют тенденцию к неограниченному росту (prompt bloat): каждая мутация добавляет правила, но редко удаляет. Length penalty (0.02 за каждые 100 токенов сверх 2000) создаёт эволюционное давление в сторону компактности. В эксперименте v3 видно, как промпт вырос до ~10K символов, а затем сократился до ~4K благодаря этому механизму.

### 6.7 Explicit Metrics in System Prompt

Формула фитнеса и описание всех метрик явно включены в system_message для эволютора. Это позволяет LLM-мутатору **понимать**, что именно оптимизируется, и принимать более осознанные решения при мутации.

### 6.8 Per-Cycle Test Evaluation

Каждый цикл оценивается на test-сплите. Эти результаты **не влияют на эволюцию** (избегаем data leakage), но позволяют мониторить реальную генерализацию в реальном времени и строить графики прогресса.

### 6.9 Diagnostic Logging

Детальная трассировка в JSONL: каждое событие (init, baseline_test, cycle_start, evolution_start/done, archive_snapshot, reclassify, validation, consolidation_start/eval/decision, expansion, refresh, cycle_test, final_test) записывается с timestamps и полными метриками. Позволяет полностью реконструировать историю эксперимента.

### 6.10 Визуализация

Система генерирует три набора графиков:
- **active_evolution_curves.png**: Validation метрики по циклам (R_global, Acc_Hard, combined_score, MAE, kappa, prompt length) с маркерами consolidation/expansion.
- **active_evolution_test.png**: Test метрики по циклам с baseline reference.
- **active_evolution_secondary.png**: Диагностические графики (batch size, pool composition, evo_best_score).
- **active_evolution_test_report.png**: Упрощённые 2×2 графики ключевых test-метрик для отчётов.

## 7. Анализ результатов и выводы

### 7.1 Что работает

1. **Активное обучение действительно фокусирует эволюцию**: Acc_Hard на тесте вырос с 30.56% до 37.66% (+7.1 п.п.) к циклу 9.
2. **Консолидация с gate эффективна**: 6 из 10 консолидаций приняты; цикл 9 показал +0.138 прирост от консолидации.
3. **Per-cycle test tracking** показал, что лучший момент — цикл 9 (test_combined = 0.853), что выше baseline (0.850).
4. **Length penalty работает**: промпт сократился с пикового ~10K до финального ~4K символов.
5. **Anchor accuracy стабильна**: 100% на всех циклах — эволюция не ломает «лёгкие» случаи.

### 7.2 Проблемы и ограничения

1. **Финальный combined_score по test (0.845) ниже baseline (0.850)**: несмотря на рост Acc_Hard, общий combined_score слегка упал из-за снижения R_worst и kappa. Однако per-cycle test на цикле 9 показывает 0.853 — парадокс объясняется различием в промптах (consolidated vs final evaluation).
2. **evo_best_score деградирует**: с 0.722 (цикл 0) до 0.686 (цикл 9). Это связано с тем, что по мере переклассификации батч становится «сложнее» — оставшиеся Hard-примеры действительно самые трудные.
3. **Нестабильность R_worst**: колеблется от 44% до 59% на тесте. Fairness-метрика чувствительна к отдельным пользователям.
4. **Prompt bloat на ранних циклах**: рост до 83 правил на цикле 6 прежде чем length penalty начал действовать.
5. **Высокий расход токенов**: ~6.8M токенов за 10 циклов. Основной расход — на воркеров (evaluation), не на мутацию.

### 7.3 Ключевые наблюдения

1. **Hard/Anchor динамика**: по мере эволюции Hard count в train уменьшается (с 56 до 47), Anchor растёт (с 24 до 33). Промпт действительно «решает» сложные случаи.
2. **Validation vs Test расхождение**: лучший val момент (цикл 7, 0.850) не совпадает с лучшим test моментом (цикл 9, 0.853). Это нормально — validation и test содержат разных пользователей.
3. **Consolidation чаще отклоняется на поздних циклах**: из первых 5 консолидаций 4 приняты; из последних 5 — только 2. Промпт становится зрелым, и консолидация чаще вносит регрессии.
4. **Soft expansion сработал один раз** (цикл 7): добавлено 15 примеров из Unseen, что помогло эволюции получить свежие данные после стагнации.

### 7.4 Итог

Система активного обучения с эволюционной оптимизацией промптов **работает**, но находится на грани значимого улучшения. Основные достижения:

- **+4.5 п.п. Acc_Hard** на test (с 30.6% до 35.1%) — существенное улучшение на сложных случаях.
- **Стабильная 100% Acc_Anchor** — нет регрессии на лёгких примерах.
- **Лучший per-cycle test combined score (0.853)** превышает baseline (0.850) на +0.003.
- **Компактный финальный промпт** (~985 токенов vs стартовый ~1207) при лучшем Acc_Hard.

Общий вердикт по финальной оценке — **DEGRADED** (−0.004 combined), что вызвано небольшим снижением R_worst и kappa. Однако per-cycle результат цикла 9 показывает потенциал системы при правильном выборе лучшего checkpoint.

## 8. Возможные направления развития

1. **Checkpoint selection по validation**: использовать best_val_prompt (цикл 7) для финальной оценки вместо последнего промпта.
2. **Увеличение числа итераций эволюции**: 30 итераций может быть недостаточно; прогоны с 50–100 итерациями могут давать лучшие результаты.
3. **Curriculum learning**: начинать с лёгких примеров и постепенно усложнять, вместо fixed 70% hard ratio.
4. **Multi-objective Pareto**: вместо скалярного fitness использовать Pareto-оптимизацию по (Acc_Hard, R_worst, length).
5. **Адаптивные веса fitness**: изменять веса в формуле fitness по мере прогресса (больше акцент на kappa после достижения Acc_Hard плато).
6. **Chain-of-Thought в промпте**: позволить воркерам «думать» перед ответом (при увеличении max_tokens) для сложных случаев.
7. **Data augmentation**: генерировать синтетические примеры для недопредставленных категорий.
8. **Более агрессивный refresh**: увеличить refresh_per_cycle для ускорения приращения разнообразия.
