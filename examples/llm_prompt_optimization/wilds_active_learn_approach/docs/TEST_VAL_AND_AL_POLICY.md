# Train / Val / Test и политика утечек

## Назначение сплитов

- **Train-пул**: источник Seen/Unseen, активного батча (Hard/Anchor), refresh/expansion.
- **Validation**: оценка лучшего промпта после эволюции; **consolidation gate**; отбор **`best_val_prompt`**; триггеры мягкого расширения, завязанные на val.
- **Test**: **не должен** влиять на выбор промпта, консолидацию, подмешивание примеров в цикл обучения или целевую эволюцию.

## Реализация в `active_loop.py` (докстринг и логика)

Из заголовка модуля:

> Holdout test: baseline + final run, plus **per-cycle test metrics** written to the log **for graphs only** (not used for consolidation, expansion triggers, or prompt selection).

Итого:

- **Per-cycle test** пишется в `active_loop_log.json` как `test_*` для визуализации (`visualize.py`).
- **Консолидация**, **soft expansion**, **выбор финального промпта для следующего цикла** опираются на **train+val** (и внутренний батч), а не на test.

## Почему это важно

Если бы test использовался для принятия решений (принять консолидированный промпт, расширить пул, ранжировать кандидатов), метрики на test стали бы оптимизируемыми и перестали бы отражать честную генерализацию.

## Baseline и final test

- **Baseline test**: стартовый промпт на test — точка отсчёта.
- **Final test**: итоговый промпт после полного прогона (или выбранный checkpoint).

Сравнение **per-cycle test** vs **final test**: на одном цикле промпт может быть «лучшим по test в логе», а финальная оценка — с другим вариантом (например после другой консолидации); это нормальный источник кажущихся противоречий в таблицах.

## Связанные эксперименты

Ветка **`baserules-consolidation`** (исторически): более жёсткая консолидация Base/Dynamic и триггеры с test — **откатили** из-за риска подгонки под holdout. Текущая политика зафиксирована в докстринге `active_loop.py` и в комментариях `config.yaml` (`active_learning`).
