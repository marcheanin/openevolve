# Расход токенов (справка)

## Откуда брать цифры

- Полный отчёт по прогону может лежать в каталоге результатов, например:  
  `results_v4_tes/token_usage_report.md`
- Токены **воркеров** доминируют: каждый отзыв = длинный системный промпт + текст отзыва × 3 модели × число оценок.
- Токены **DeepSeek R1** в OpenEvolve (мутации) и вызовы консолидации могут учитываться отдельно; в некоторых отчётах указано, что **R1 в OpenEvolve не включён** в сводку воркеров.

## Пример: `results_v4_tes/token_usage_report.md` (сводка)

| Модель | Input | Output | Total |
|--------|-------|--------|-------|
| consolidation / deepseek-r1 | ~16k | ~6k | ~22k |
| deepseek-chat-v3 | ~3.9M | ~115k | ~4.05M |
| gemma3-27b | ~4.1M | ~16k | ~4.11M |
| gpt-4o-mini | ~3.86M | ~16k | ~3.88M |
| **Итого (воркеры + указанная консолидация)** | **~11.9M** | **~154k** | **~12.1M** |

Для масштаба v3 см. раздел «Расход токенов» в [PROJECT_REPORT.md](PROJECT_REPORT.md) (~6.8M input на 10 циклов — другой прогон и методология подсчёта).

## Практический вывод

- Укорочение промпта (правила + few-shot) напрямую снижает стоимость и латентность оценки.
- `max_code_length` в `config.yaml` и `PROMPT_LEN_LIMIT` / length penalty в `evaluator.py` — рычаги против **prompt bloat**; см. [EXPERIMENTS_AND_LESSONS.md](EXPERIMENTS_AND_LESSONS.md).
