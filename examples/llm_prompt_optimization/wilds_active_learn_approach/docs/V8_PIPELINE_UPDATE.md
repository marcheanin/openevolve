# Прогон `results_v8` и обновление пайплайна (post-v8)

## Что показал v8

Запуск с **merge-first** промптами, `max_code_length=7500`, **early stopping** OpenEvolve (patience 5), **consolidation gate** только относительно val эволюции в том же цикле.

Ключевые наблюдения (см. `results_v8/active_loop_log.json`, `analyze_run.py`):

- **Пик качества на test** пришёлся на **AL cycle 3** (per-cycle test): `test_combined_score` ≈ **0.851**, `R_global` ≈ **75%**, `Acc_Hard` ≈ **36.6%**.
- **Финальный** отчётный промпт после цикла 4 оказался **слабее**: падение test combined до ~**0.747** при том, что **лучший по val** оставался на уровне цикла 1–3 (`seed_val_score` до **0.843**). Итоговые метрики «baseline → final» дали лишь **~+0.0009** по combined — в основном из-за того, что **финальный тест считался по последнему циклу**, а не по лучшему val.
- **Merge-first** в сочетании с R1 дал **компактные** промпты (~5–6k символов, без length-fail), но иногда **слишком агрессивно** схлопывал граничные правила — страдает обобщение на hard-кейсы.
- **Soft expansion** сработал на цикле 3 при **плато val в окне**, хотя текущий `seed_val_score` был **на историческом максимуме** — лишний раз расширил пул и сместил баланс Hard/Anchor.
- **Состав батча**: целевой `hard_ratio=0.7` соблюдался по счётчику в `cycle_start`; при нехватке слотов старый код **сначала добирал Anchor**, что могло **размывать** долю Hard при коротком первичном отборе — исправлено (см. ниже).

## Что изменили в коде и конфиге (ожидаемый эффект — «v9-логика»)

1. **Финальный holdout test** считается по промпту с **лучшим `seed_val_score` на validation** (`best_val_prompt`), а не по последнему циклу. В артефактах: `final_prompt.txt` = этот промпт, `last_cycle_prompt.txt` = последний цикл, `best_val_meta.json` — метаданные.
2. **Раннее завершение AL**: `al_early_stopping_patience` / `al_early_stopping_min_cycles` — остановка, если несколько циклов подряд **нет нового лучшего val** (меньше «дожимать» слабые хвостовые циклы).
3. **Смягчение merge-давления** в `system_message` и `CONSOLIDATION_SYSTEM`: ориентир **30–50** правил, явный запрет **сливать разные boundary** в одну размытую строку.
4. **Второй gate консолидации**: отклонение, если `cons_val` заметно ниже **ref_best = max(исторический best val, val эволюции в этом цикле)** (`consolidation_gate_vs_best_delta`).
5. **Soft expansion**: не запускать **только soft**-expansion, если текущий `seed_val` **на пике** относительно лучшего val (`soft_expansion_skip_near_best`).
6. **Active batch**: при доборе до `batch_size` сначала **оставшиеся Hard**, потом Anchor; предупреждение, если доля Hard в батче ниже `min_hard_batch_ratio` (при возможности пула).

## Чего ждём от следующего полного прогона

- Отчётный **final test** ближе к **лучшему per-cycle test**, если val-коррелирует с test (как в v8 на цикле 3).
- Меньше **регрессий после консолидации**, когда цикл уже слабый относительно прошлого лучшего val.
- Меньше **лишних расширений пула** сразу после нового максимума val.
- Более стабильная **доля Hard** в батче при нехватке слотов после первичного отбора.

Точные цифры — только после нового run в отдельной папке `results_*`; этот файл — зафиксированный контекст гипотез и правок.
