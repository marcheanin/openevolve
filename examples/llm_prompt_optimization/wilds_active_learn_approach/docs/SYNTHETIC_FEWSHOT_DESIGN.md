# Синтетические граничные FewShotExamples

> На основе идей из статьи *Intent-based Prompt Calibration: Enhancing prompt
> optimization with synthetic boundary cases* (Levi, Brosh, Friedmann — ICLR 2024).

## 1. Проблема

### 1.1 Текущее состояние

Секция `<FewShotExamples>` промпта формируется из **реальных отзывов** датасета
(изначально 7 hand-crafted примеров в `initial_prompt.txt`). При эволюции
мутатор может заменять примеры на другие из error artifacts, но источник всегда
один — реальные тексты из train pool.

### 1.2 В чём проблема

1. **Датасет ограничен.** В train pool ~513 примеров. Для конкретной confusion
   pair (например, gold=4 → pred=5) может не найтись отзыва, который *идеально*
   демонстрирует границу между этими классами.

2. **Реальные отзывы не образуют минимальных пар.** Отзыв на 4 и отзыв на 5
   из датасета отличаются по теме, длине, стилю, автору — модель не видит,
   какое *именно* свойство текста отличает 4 от 5. Полезный few-shot должен
   показать: «вот два почти одинаковых отзыва, один — 4, другой — 5, и разница
   в _одном конкретном сигнале_».

3. **Acc_Hard — главный bottleneck.** Во всех пост-v8 прогонах test Acc_Hard
   застрял на уровне 0.24 (при val ~0.36). Top confusion pairs (4↔5, 3↔4,
   2↔3) составляют ~60–70% ошибок. Текущие few-shot не адресуют эти границы
   таргетированно.

4. **Мутатор работает вслепую.** Он получает ошибки и может вставить реальный
   отзыв из artifacts в `<FewShotExamples>`, но у него нет инструмента для
   создания *контрастных пар*, которые явно обучают модель граничным различиям.

### 1.3 Что говорит статья

AutoPrompt (Levi et al.) показал, что **синтетические граничные примеры**
эффективнее реальных данных для калибровки промпта: LLM генерирует трудные
кейсы, целенаправленно тестирующие слабые места. Разница с нашим применением:
в AutoPrompt синтетика используется для *диагностики* (тестирование модели),
мы используем её для *обучения* (few-shot демонстрации внутри промпта).

## 2. Предлагаемое решение

### 2.1 Идея

Добавить **промежуточный LLM-вызов** (Synthetic FewShot Generator) между
циклами или перед каждой эволюцией. Генератор получает:

- текущую confusion matrix и error artifacts (те же данные, что видит мутатор);
- текущие `<FewShotExamples>` из промпта.

На выходе — **полностью новый набор синтетических `<FewShotExamples>`**,
заменяющий оригинальные. Реальные примеры из датасета больше не используются
в few-shot секции.

### 2.2 Почему только синтетические

- **Контроль над контрастом.** Генератор создаёт *пары-близнецы*: два отзыва
  с минимальным различием, один на N звёзд, другой на N+1. Это невозможно
  гарантировать с реальными отзывами.
- **Адаптивность.** Каждый цикл генерируются примеры, нацеленные на *текущие*
  confusion pairs. Если на цикле 3 главная проблема — 4→5, примеры будут про
  4 vs 5; если к циклу 5 проблема сместится к 2→3, примеры перестроятся.
- **Чистота эксперимента.** Смешивание реальных и синтетических примеров
  затрудняет интерпретацию: непонятно, какой тип помогает. Полный переход
  на синтетику позволяет чётко измерить эффект.

### 2.3 Что делает мутатор (эволюция)

Мутатор **больше не трогает `<FewShotExamples>`** — эта секция контролируется
исключительно Synthetic FewShot Generator. Мутатор фокусируется на
`<DynamicRules>` и `<BaseGuidelines>`, как и раньше, но теперь few-shot
секция гарантированно содержит таргетированные примеры под текущие ошибки.

## 3. Архитектура

### 3.1 Новый компонент: `SyntheticFewShotGenerator`

```
active_loop.py
  │
  ├── [ПЕРЕД ЭВОЛЮЦИЕЙ] SyntheticFewShotGenerator
  │     Вход:
  │       - confusion matrix (из error artifacts текущего батча)
  │       - текущие <FewShotExamples> из промпта
  │       - текущие <DynamicRules> (для контекста)
  │       - топ ошибки: MISCLASSIFIED + BORDERLINE примеры
  │     LLM-вызов → deepseek-r1 (или тот же эволютор)
  │     Выход:
  │       - новый блок <FewShotExamples> (7-12 синтетических примеров)
  │     Промпт с новыми FSE подставляется как seed для эволюции
  │
  ├── OpenEvolve (эволюция DynamicRules + BaseGuidelines)
  │     Мутатор НЕ меняет <FewShotExamples>
  │     ...
```

### 3.2 Поток данных (один AL цикл)

```
1. build_active_batch()          → batch (Hard + Anchor)
2. initial eval на batch          → confusion matrix, error artifacts
3. ★ SyntheticFewShotGenerator   → новые <FewShotExamples> в промпте
4. OpenEvolve (мутирует только DynamicRules/BaseGuidelines)
5. reclassify, validation, consolidation, expansion — без изменений
```

### 3.3 Промпт для генератора

Генератор получает system message примерно следующей структуры:

```
Ты — эксперт по созданию обучающих примеров для классификации отзывов
Amazon Home & Kitchen по шкале 1–5 звёзд.

Тебе даны:
1. Текущие FewShotExamples из промпта.
2. Confusion matrix: какие пары рейтингов путаются чаще всего.
3. Конкретные примеры ошибок (MISCLASSIFIED) и пограничных случаев (BORDERLINE).
4. Текущие DynamicRules (для понимания, какие границы уже описаны).

Задача: создать НОВЫЙ набор из 7–12 FewShotExamples.

ПРИНЦИПЫ ГЕНЕРАЦИИ:

1. **Контрастные пары.** Для каждой top confusion pair (напр. gold=4→pred=5)
   создай пару синтетических отзывов-близнецов:
   - один с рейтингом N (нижний класс пары)
   - другой с рейтингом N+1 (верхний класс пары)
   Отзывы должны быть МАКСИМАЛЬНО ПОХОЖИ по теме и стилю, отличаясь
   ОДНИМ ключевым сигналом (напр.: наличие/отсутствие рекомендации,
   упоминание возврата, тон заключительного предложения).

2. **Покрытие всех классов.** Итоговый набор должен содержать хотя бы
   один пример для каждого рейтинга 1–5.

3. **Реалистичность.** Отзывы должны выглядеть как настоящие Amazon-отзывы
   на товары Home & Kitchen: посуда, текстиль, мебель, бытовая техника.
   Длина 2–5 предложений.

4. **Приоритет top confusion pairs.** Если top-3 ошибки — это 4→5, 3→4, 5→4,
   то большинство примеров должны демонстрировать эти границы.

ФОРМАТ ВЫВОДА:

<FewShotExamples>

Example 1:
Review: [синтетический отзыв]
Rating: [1-5]

Example 2:
...

</FewShotExamples>
```

## 4. Детали реализации

### 4.1 Файлы

| Файл | Изменение |
|------|-----------|
| `synthetic_fewshot_generator.py` | **Новый.** Класс `SyntheticFewShotGenerator`: LLM-вызов, парсинг, валидация. |
| `active_loop.py` | Вызов генератора между шагами 2 и 4 (перед эволюцией). Подстановка новых FSE в seed prompt. |
| `evaluator.py` | Без изменений (artifacts уже содержат confusion matrix и примеры ошибок). |
| `config.yaml` | Новая секция `synthetic_fewshot`. |

### 4.2 Конфигурация

```yaml
active_learning:
  synthetic_fewshot:
    enabled: true
    n_examples: 10                  # целевое число примеров
    n_boundary_pairs: 3             # сколько контрастных пар для top confusion
    model: "deepseek/deepseek-r1"   # модель генератора (по умолчанию = эволютор)
    temperature: 0.7
    max_tokens: 4096
    max_retries: 2
```

### 4.3 Валидация сгенерированных примеров

После генерации проверяем:
- XML-структура: `<FewShotExamples>` открыт и закрыт.
- Каждый пример содержит `Review:` и `Rating: [1-5]`.
- Покрытие: хотя бы один пример на каждый рейтинг 1–5.
- Количество: 7–12 примеров.

Если валидация провалена — retry (до `max_retries`), при полном
провале — оставить текущие FSE без изменений.

### 4.4 Защита от регрессии

Синтетические FSE подставляются в seed prompt ПЕРЕД эволюцией.
Эволюция оценивает промпт с новыми FSE на active batch. Если промпт
с новыми FSE хуже, чем с предыдущими — эволюция сама отберёт лучший
вариант через MAP-Elites (initial program оценивается первым).

Дополнительный gate не нужен: MAP-Elites уже обеспечивает, что
финальный best program не хуже seed.

### 4.5 Запрет мутации FewShotExamples

В system_message мутатора добавить:

```
<FewShotExamples> are pre-generated for this cycle and must NOT be
modified. Focus your changes on <DynamicRules> and <BaseGuidelines>.
```

Альтернатива (мягче): оставить мутатору право корректировать FSE,
но тогда синтетические примеры служат лишь стартовой точкой.
Рекомендация: начать с жёсткого запрета для чистоты эксперимента.

## 5. Ожидаемый эффект

| Метрика | Текущее (v10) | Гипотеза | Механизм |
|---------|---------------|----------|----------|
| Acc_Hard (test) | ~0.24 | +5–10 п.п. | Контрастные пары напрямую атакуют top confusion pairs |
| Val-test gap | до 0.07 | Снижение ~0.02 | Синтетические примеры обобщают паттерн, не привязаны к конкретным текстам |
| combined_score (test) | 0.812 | +0.01–0.03 | Через рост Acc_Hard без регрессии Anchor |
| Стагнация эволюции | best к итерации 4–6 | Позже | Мутатор стартует с лучшего seed (таргетированные FSE) |

### Риски

- **Нереалистичность.** Если синтетические отзывы выглядят искусственно, воркеры
  могут научиться «артефактам» стиля. Митигация: в промпте генератора требовать
  реалистичность и разнообразие стиля.
- **Prompt bloat.** 10 синтетических примеров с reasoning могут быть длиннее
  7 текущих. Митигация: лимит длины примера (3–5 предложений), length_penalty
  в fitness.
- **Стоимость.** +1 LLM-вызов на генерацию FSE per cycle. При ~4K tokens
  in + ~3K tokens out на вызов R1 — пренебрежимо на фоне ~1.7M worker tokens.

## 6. План экспериментов

1. **v12-baseline**: прогон с текущим конфигом v11 для свежего baseline.
2. **v12-synth**: синтетические FSE включены, мутатор не трогает FSE.
   Сравнение с v12-baseline по Acc_Hard, combined, val-test gap.
3. **v12-synth-free**: синтетические FSE как стартовая точка, мутатор
   может корректировать. Сравнение с v12-synth для оценки: лучше ли
   фиксированные FSE или мутатор может их улучшить.

## 7. Связь со статьёй (для отчёта)

В AutoPrompt синтетические граничные примеры — инструмент **диагностики**:
они подаются модели для выявления слабых мест, человек аннотирует спорные
случаи. В нашем фреймворке синтетические примеры переносятся из контура
оценки в контур **генерации промпта**: они не тестируют модель, а учат её
проводить границу между соседними классами через few-shot демонстрации.

Отбор и компоновка примеров происходят автоматически, без участия
человека-аннотатора. Confusion matrix из ансамблевой оценки заменяет
ручную аннотацию, а эволюционный цикл обеспечивает итеративное уточнение.
