# Прогоны v9 и v10: итоги, сравнение, рекомендации

## Контекст

После **v8** (см. [V8_PIPELINE_UPDATE.md](V8_PIPELINE_UPDATE.md)) были реализованы 5 изменений пайплайна:
1. Финальный тест по `best_val_prompt` вместо last-cycle.
2. AL-level early stopping (`al_early_stopping_patience`).
3. Смягчение merge-давления (бюджет 30-50 правил вместо 25-40).
4. Двойной consolidation gate (vs evo-val + vs global best val).
5. Guard на soft expansion при val-пике, приоритет Hard в batch.

**v9** — первый прогон с этими изменениями.
**v10** — изменён только **system_message мутатора**: переписан от жёстких правил к свободным ориентирам.

---

## v9 — результат изменений пайплайна

### Конфигурация
- `system_message`: жёсткий MERGE-FIRST, бюджет 30-50, MANDATORY coverage, процедурные инструкции.
- `al_early_stopping_patience: 2`, `al_early_stopping_min_cycles: 2`.
- `consolidation_gate_vs_best_delta: 0.04`, `soft_expansion_skip_near_best: 0.005`.
- `temperature` workers: 0.1.

- Консолидация: цикл 0 **rejected**, циклы 1-2 accepted.

### Диагноз

Изменения пайплайна (gates, early stop, best_val_prompt) **работали корректно**: early stop сэкономил ~5000s и ~7M токенов, best_val_prompt поднял final test с ~0.78 до 0.806. Но **модель-мутатор не справилась**: жёсткие инструкции (MERGE-FIRST, MANDATORY, бюджет) ограничивали креативность; R1 не мог найти улучшения поверх уже оптимизированного промпта.

## v10 — эффект свободного системного промпта

### Единственное изменение

`system_message` мутатора полностью переписан: от "MUST / MANDATORY / CRITICAL" к "considerations, not rigid rules". Модели дана свобода:
- Реструктурировать `<DynamicRules>` как угодно (списки, таблицы, деревья решений).
- Самостоятельно решать, сколько правил нужно (без фиксированного бюджета).
- Фокусироваться на гипотезах по ошибкам, а не на механическом применении процедур.
- Описана логика fitness и вес метрик, чтобы модель понимала, что оптимизирует.

- 10 diverse few-shot examples, покрывающих все 5 уровней.

## Ключевые наблюдения и открытые проблемы

### 1. Шум оценки (temperature workers = 0.1)

При текущей `temperature: 0.1` один и тот же промпт даёт разброс ~**0.04** по combined_score между оценками. Это сопоставимо с:
- Порогами consolidation gate (0.02-0.04).
- Типичным delta parent→child в эволюции (0.01-0.06).
- Разницей между целыми экспериментами (v9 vs v10 = 0.006).

**Влияние**: consolidation gate может случайно принять/отклонить; эволюция может выбрать "улучшение", которое просто повезло; best_val_prompt может не быть реально лучшим.

**Рекомендация**: `temperature: 0.0` для workers. Ансамбль из 3 разных моделей уже обеспечивает естественное разнообразие — температура добавляет чистый шум поверх архитектурных различий.

### 2. Val-test gap

| Цикл v10 | val_combined | test_combined | gap |
|---|---|---|---|
| 0 | 0.803 | 0.785 | 0.018 |
| 1 | 0.818 | 0.770 | 0.048 |
| 2 | 0.800 | 0.754 | 0.046 |
| 3 | 0.841 | 0.772 | 0.069 |
| 4 | 0.820 | 0.810 | 0.010 |

Gap до 0.07 — часть из-за шума (температура), часть из-за overfitting к 181-example val set. Особенно заметно на Acc_Hard: val = 0.36, test = 0.24.

**Рекомендация**: увеличение `max_val_users` (с 15 до 20-25) даст больше validation примеров и уменьшит val-test gap.

### 3. Acc_Hard на тесте не растёт

Во всех пост-v8 прогонах test Acc_Hard = 0.24-0.28, при val Acc_Hard = 0.30-0.38. Эволюция учит "трудные" паттерны из train/val, но test-hard отличаются. v4_tes достигал 0.395 — значит потенциал есть, но текущий подход не выжимает его.

## Рекомендации (реализованы в v11)

Все пункты ниже применены в `config.yaml` для `results_v11`:

1. **`temperature: 0.0` для workers** — убирает шум ~0.04 из оценки. Ансамбль разнообразен за счёт 3 разных моделей.
2. **`max_val_users: 22`** (было 15) — больше val-примеров → меньше val-test gap.
3. **`max_iterations: 500`** (было 100) — эволюцию ограничивает только `early_stopping_patience: 5`. В v10 модель находила улучшения на поздних итерациях; теперь потолка нет.
4. **BaseGuidelines разрешён к правке** — в system_message снято ограничение "Keep BaseGuidelines unchanged"; модель может уточнять формулировки (но обязана сохранить scale и output format).
5. **`consolidation_gate_delta: 0.01`** (было 0.02) — при temperature=0.0 шум мал, gate может быть чувствительнее.

## v11 — план эксперимента

### Изменения конфига

| Параметр | v10 | v11 | Причина |
|---|---|---|---|
| `worker temperature` | 0.1 | **0.0** | Убрать шум оценки |
| `max_val_users` | 15 | **22** | Больше val → меньше gap |
| `max_iterations` | 100 | **500** | Только early stop ограничивает эволюцию |
| BaseGuidelines | frozen | **editable** | Модель уже правила в v10 — легализовать |
| `consolidation_gate_delta` | 0.02 | **0.01** | Чувствительнее при меньшем шуме |

Всё остальное (system_message, models, fitness, AL params) — без изменений от v10.

### Гипотезы

1. **Детерминированная оценка** сделает fitness-сравнения надёжными: эволюция не будет «терять» хорошие промпты из-за случайного занижения, а consolidation gate перестанет пропускать/блокировать промпты по шуму.
2. **Больший val set** уменьшит overfitting к val-специфичным паттернам — val_Acc_Hard и test_Acc_Hard должны сблизиться.
3. **Неограниченные итерации** дадут модели возможность продолжать поиск пока есть прогресс. В v10 early stop срабатывал после 6-12 итераций — теперь если модель может найти ещё, ей ничего не мешает.
4. **Легальная правка BaseGuidelines** позволит модели добавлять Key Principles без нарушения инструкций — меньше "конфликт" между инструкцией и тем, что модель считает нужным.

### Ожидания

- `combined_score` ≥ 0.82 (при удачном раскладе — приблизиться к v4_tes = 0.862).
- Val-test gap < 0.03 (было до 0.07 в v10).
- Стабильные fitness-сравнения (одинаковый промпт = одинаковый скор).
- Больше итераций за цикл (потенциально 15-30 при patience=5).
