## 2. Сравнение v6 vs `results_v4_tes` (AL iter 0) — диагноз

### 2.1 Стагнация OpenEvolve (v6)

- Лучшая программа на активном батче часто достигается **раньше конца** заданного числа итераций (пример: лучший скор на **итерации 4** из **20**; дальнейшие итерации не улучшают best в checkpoint-логах).
- **Интерпретация**: при фиксированном батче и текущем мутаторе популяция быстро упирается в локальный оптимум; увеличение итераций без смены давления на длину/разнообразие может не окупаться.

### 2.2 Фитнес на батче (Hard-фокус)

- У **v4_tes** best по эволюции на iter 0: **combined_score** выше, **Acc_Hard** выше (порядок **~0.73 / ~0.54** vs **~0.69 / ~0.46** у v6 — ориентировочно, уточнять по JSON).
- **Интерпретация**: v6 хуже «подстроился» под тот же тип батча при сопоставимом пайплайне.

### 2.3 Prompt bloat (длина и специфичность правил)

- **Длина**: best-промпт v6 существенно длиннее, чем у v4_tes (порядок **~7500+ символов** vs **~4800+** для сравнимых файлов `best_prompt.txt`).
- **DynamicRules** в v6 чаще содержат **узкие, примеро-специфичные** формулировки; это коррелирует с ростом длины и может ухудшать обобщение при том же length penalty.
- **Связь с фитнесом**: штраф за длину в `evaluator.py` (`PROMPT_LEN_LIMIT = 2000` токенов) может быть **недостаточно жёстким** относительно `max_code_length: 10000` в `config.yaml` — модель «влезает» в лимит символов, но платит умеренный штраф.

### 2.4 Consolidation gate

- **v4_tes**: консолидация на iter 0 могла быть **отклонена** (падение val combined **> consolidation_gate_delta** ~0.02 относительно evo val).
- **v6**: консолидация **принята** при **малой** дельте val (< 0.02), при этом **абсолютный** evo val мог быть ниже, чем у v4_tes.
- **Интерпретация**: gate защищает от **внезапной** деградации консолидации относительно **текущего** evo-промпта, но **не** ранжирует прогоны между собой; низкий evo val + «безопасная» консолидация всё равно дают слабую стартовую точку для следующего цикла.

### 2.5 Недетерминизм baseline test

- Небольшие расхождения **baseline_test** между прогонами возможны из-за ансамбля и API (temperature воркеров низкая, но полный детерминизм не гарантирован).
- Для сравнения прогонов опираться на **относительные** метрики внутри одного run и на **val**, а не только на одиночный замер test.

## 3. Проблемы длины и эволюции (сводка)

| Проблема | Проявление | Рычаги |
|----------|------------|--------|
| **Prompt bloat** | Рост `<DynamicRules>` и few-shot; много дублей и edge-case правил. | Сильнее length penalty; ниже `max_code_length`; в system_message усилить «merge/prune»; MAP-Elites уже давит через ось `prompt_length`. |
| **Ранняя стагнация** | Best score не растёт после N итераций. | Меньше итераций на цикл при том же батче; смена temperature/top-k; refresh/expansion чаще; другой активный батч. |
| **Несоответствие символов и токенов** | Лимит 10000 символов в конфиге vs штраф по токенам. | Выровнять политику (например явный лимит токенов в мутации или жёсткий отказ оценки). |
| **Консолидация без выигрыша** | Val не падает сильно, но и не чинит bloat. | Отдельный промпт консолидации «сжать без потери val»; двухшаговая консолидация. |

## 4. Что уже пробовали (концептуально)

- **Прямая эволюция без AL** (ранние Exp 1–3b в `wilds_experiment`): прирост есть, быстрое насыщение.
- **AL v2**: без полного набора защит — деградация val.
- **AL v3** (см. отчёт): batch 80, hard 70%, MAP-Elites по длине × Acc_Hard, length penalty, consolidation gate, refresh, soft/hard expansion, rule coverage, per-cycle test **только для графиков**.
- **Ужесточение консолидации Base/Dynamic + test-триггеры** (ветка **baserules-consolidation**): откат из-за **утечки test** в решения; текущая политика — [TEST_VAL_AND_AL_POLICY.md](TEST_VAL_AND_AL_POLICY.md).
- **v4_tes / v5 / v6**: вариации прогонов для сравнения консолидации, длины и числа итераций эволюции.

## 5. Рекомендации для следующих итераций

1. **Давление на длину**: увеличить штраф или снизить `max_code_length`; явно измерять токены best в логе цикла.
2. **Мутатор**: в `system_message` приоритизировать **слияние** похожих правил и **удаление** слабых дубликатов, а не только добавление.
3. **Итерации эволюции**: если best стабилизируется к середине цикла — сократить `max_iterations` или ввести раннюю остановку по плато **combined_score** на батче.
4. **Сравнение прогонов**: фиксировать seed/API-версии где возможно; логировать длину в токенах и символах для каждого `best_prompt`.
5. **Выбор промпта**: использовать **`best_val_prompt`** для финальной оценки, если последний цикл деградировал (уже описано в отчёте v3). **Реализовано в коде** после v8: `final_prompt.txt` = лучший по `seed_val_score` на val; см. [V8_PIPELINE_UPDATE.md](V8_PIPELINE_UPDATE.md).
